# Design Doc — DD-00002 Administración de Usuarios y Roles

## 1. Contexto y objetivo
Este módulo define las capacidades para que un **Administrador** gestione los usuarios del sistema Gestiora de forma segura, auditable y coherente con el modelo de autenticación y sesiones definido en **DD-00001**.

El objetivo es:
- Controlar quién puede acceder al sistema.
- Gestionar roles y estados de los usuarios.
- Garantizar que los cambios tengan efecto inmediato (revocación de sesiones).
- Mantener trazabilidad completa de las acciones administrativas.

**Fuera de alcance (v1):**
- Recuperación de contraseña por el usuario.
- Cambio de contraseña sin intervención administrativa.
- Gestión avanzada de permisos (ABAC).
- Delegación de administración.

---

## 2. Alcance funcional
Cubre **Épica B — Administración de Usuarios y Roles**:

- US-B01 Creación de usuario
- US-B02 Consulta y listado de usuarios
- US-B03 Modificación de usuario
- US-B04 Activación y desactivación
- US-B05 Eliminación lógica (soft delete)
- US-B06 Revocación de sesiones

### Roles implicados
- **Administrador** (único rol autorizado)
- Usuario (sujeto gestionado)

---

## 3. Modelo de dominio

### 3.1 Entidades

**Usuario**
- uuid
- mail (único)
- nombre
- avatar (opcional)
- passwordHash
- status: ACTIVO | INACTIVO | ELIMINADO
- createdAt
- updatedAt
- deletedAt (nullable)

**Rol**
- id
- nombre (Usuario | Administrador)

**UserRole**
- userId
- roleId

**Sesion**
- (ver DD-00001)
- asociada al usuario

**EventoAuditoria**
- (transversal, compartido)

---

### 3.2 Invariantes de dominio
- Un usuario debe tener **al menos un rol**.
- Un usuario ELIMINADO no puede:
  - autenticarse
  - ser reactivado
- Un usuario INACTIVO:
  - no puede iniciar sesión
  - mantiene su historial
- No se permite eliminar físicamente usuarios.
- Un administrador **no puede eliminarse a sí mismo** (regla defensiva).

---

## 4. Casos de uso (Application Layer)

### UC-01 Crear usuario
**Input:** mail, password inicial, roles, estado inicial  
**Output:** userId  

**Reglas:**
- Solo Administrador.
- Validar unicidad de mail.
- Hashear contraseña antes de persistir.
- Asignar rol inicial.
- Usuario creado como ACTIVO por defecto (configurable).
- Auditar acción.

**Errores:**
- `USER_ALREADY_EXISTS`
- `FORBIDDEN`

---

### UC-02 Listar usuarios
**Input:** filtros (estado, rol), paginación  
**Output:** lista de usuarios (sin credenciales)

**Reglas:**
- Solo Administrador.
- No exponer passwordHash, tokens, sesiones.
- Soportar paginación y ordenación.

---

### UC-03 Consultar usuario
**Input:** userId  
**Output:** detalle del usuario  

**Reglas:**
- Solo Administrador.
- 404 si no existe o eliminado.

---

### UC-04 Modificar usuario
**Input:** userId, campos modificables  
**Campos permitidos:**
- roles
- estado (ACTIVO / INACTIVO)
- nombre / avatar

**Reglas:**
- No modificar identificadores internos.
- Cambio de roles tiene efecto inmediato.
- Auditar antes/después.

---

### UC-05 Activar / desactivar usuario
**Input:** userId, estado  
**Output:** ok  

**Reglas:**
- Desactivar:
  - revoca sesiones activas
  - impide nuevos logins
- Activar:
  - permite login nuevamente
- No permitido sobre usuarios ELIMINADOS.

---

### UC-06 Eliminación lógica (soft delete)
**Input:** userId  
**Output:** ok  

**Reglas:**
- Marcar como ELIMINADO.
- Revocar todas las sesiones.
- Excluir de listados por defecto.
- Mantener trazabilidad histórica.
- No reversible.

---

### UC-07 Revocar sesiones
**Input:** userId  
**Output:** ok  

**Reglas:**
- Revoca todas las sesiones activas.
- Invalida refresh tokens.
- No elimina sesiones históricas.

## 5. API / Interfaces

### 5.1 Endpoints propuestos (admin)

- `POST   /admin/users`
- `GET    /admin/users`
- `GET    /admin/users/{userId}`
- `PUT    /admin/users/{userId}`
- `PATCH  /admin/users/{userId}/status`
- `DELETE /admin/users/{userId}` (soft delete)
- `POST   /admin/users/{userId}/sessions/revoke`

### 5.2 Seguridad
- Todos los endpoints:
  - Access token válido
  - Rol Administrador
- Responder 403 si no autorizado.

### 6.1 Tablas

- `user_roles`

### 6.2 Estrategia de borrado
- Soft delete:

## 7. Seguridad y control de acceso

- Enforcement centralizado (middleware/policy).
- Claims de rol presentes en access token.
- Revalidación de permisos en cada request.
- Revocación inmediata vía sesiones.

---

## 8. Observabilidad y auditoría

### Eventos mínimos
- USER_CREATED
- USER_UPDATED
- USER_ACTIVATED
- USER_DEACTIVATED
- USER_DELETED
- USER_ROLE_CHANGED
- USER_SESSIONS_REVOKED

Cada evento incluye:
- actorAdminId
- targetUserId
- timestamp
- resultado

---

## 9. Decisiones de diseño (ADR-style)

### D1 — Administración centralizada por rol
**Decisión:** solo Administrador puede gestionar usuarios.  
**Consecuencia:** simplicidad y menor superficie de ataque.

### D2 — Soft delete obligatorio
**Decisión:** nunca borrar usuarios físicamente.  
**Consecuencia:** trazabilidad completa y cumplimiento.

### D3 — Revocación de sesiones automática
**Decisión:** cambios críticos revocan sesiones.  
**Consecuencia:** seguridad inmediata, ligera carga extra.

---

## 10. Riesgos y evolución futura

### Riesgos
- Errores en revocación → sesiones válidas indebidamente.
- Falta de auditoría detallada.
- Crecimiento del modelo de permisos sin refactor.

### Evolución
- Delegación de administración.
- Permisos por recurso.
- Gestión de contraseñas por el usuario.
- MFA.
