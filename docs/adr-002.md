# ADR-0002: Patron de Transacciones con Rollback para Tests de Postgres

## Estado

Aceptado

## Fecha

2026-02-18

## Contexto

Los tests de integracion con PostgreSQL (Supabase) presentaban varios problemas:

1. **Datos residuales**: Los tests dejaban datos en la base de datos despues de ejecutarse
2. **Conflictos entre tests**: Tests que dependian de un estado limpio fallaban si otro test habia dejado datos
3. **No reproducibles**: Ejecutar los tests multiples veces producia resultados inconsistentes
4. **Limpieza manual**: Se requeria limpiar la base de datos manualmente entre ejecuciones

El proyecto necesitaba una estrategia que garantizara:

* Aislamiento completo entre tests
* No dejar datos residuales en la base de datos
* Tests reproducibles y deterministas
* Minimo impacto en el rendimiento

## Decision

Se adopta el **patron de transacciones con savepoint/rollback** para todos los tests de integracion con PostgreSQL.

### Implementacion

Se crea un helper compartido `createPostgresTestContext()` en `tests/shared/helpers/postgres-test-context.ts` que encapsula:

1. **Conexion a la base de datos**: Usa `DATABASE_URL` del entorno
2. **Transaccion global**: Se inicia en `beforeAll` y se revierte en `afterAll`
3. **Savepoints por test**: Cada test crea un savepoint en `beforeEach` y lo revierte en `afterEach`

### Flujo de ejecucion

```
beforeAll:
  - Conectar a PostgreSQL
  - BEGIN (transaccion global)
  - Crear tablas si no existen

beforeEach:
  - SAVEPOINT test_savepoint
  - DELETE datos (limpieza dentro de la transaccion)

afterEach:
  - ROLLBACK TO SAVEPOINT test_savepoint

afterAll:
  - ROLLBACK (transaccion global)
  - Cerrar conexion
```

### Uso en tests

```typescript
import { describe, it, expect, beforeAll, afterAll, beforeEach, afterEach } from 'vitest';
import { describeIf } from '../../../shared/helpers/describe-if.js';
import { createPostgresTestContext } from '../../../shared/helpers/postgres-test-context.js';

const DATABASE_URL = process.env.DATABASE_URL;

describeIf(DATABASE_URL)('PostgresMyRepository', () => {
  const ctx = createPostgresTestContext();

  beforeAll(async () => {
    await ctx.setup();
    // Crear tablas con CREATE TABLE IF NOT EXISTS
    await ctx.sql`CREATE TABLE IF NOT EXISTS my_table (...)`;
  });

  afterAll(async () => {
    await ctx.cleanup();
  });

  beforeEach(async () => {
    await ctx.beginTransaction();
    // Limpiar datos (dentro de la transaccion)
    await ctx.sql`DELETE FROM my_table`;
  });

  afterEach(async () => {
    await ctx.rollbackTransaction();
  });

  it('should do something', async () => {
    // Test usa ctx.sql para queries
    const repo = new PostgresMyRepository(ctx.sql);
    // ...
  });
});
```

## Consecuencias

### Positivas

* **Aislamiento total**: Cada test se ejecuta en un contexto limpio
* **Sin datos residuales**: El rollback garantiza que no quedan datos en la DB
* **Reproducible**: Los tests producen los mismos resultados cada vez
* **Rendimiento**: Rollback es mas rapido que DELETE/TRUNCATE real
* **Reutilizable**: El helper se comparte entre todos los tests de Postgres

### Negativas

* **Complejidad adicional**: Requiere entender el patron de transacciones
* **No prueba commits reales**: El codigo de produccion hace commit, los tests no

### Neutrales

* **Requiere DATABASE_URL**: Los tests se saltan si no hay conexion configurada
* **Tablas persisten**: Las tablas se crean con IF NOT EXISTS y persisten entre ejecuciones
