# Design Doc — DD-00008 Sistema de Logging Técnico

## 1. Contexto y objetivo
Gestiora necesita un sistema de logging técnico que permita registrar información
relevante durante la ejecución del sistema: arranque, operaciones, advertencias y errores.

Actualmente:
- El `AuditLogger` cubre eventos de negocio (login, CRUD, etc.).
- El logger de Fastify está acoplado a la capa HTTP.
- No existe un logger de propósito general como puerto/adaptador.

**Objetivo principal:** implementar un logger técnico siguiendo Clean Architecture,
inyectable en cualquier capa, sencillo pero escalable.

**Fuera de alcance (por ahora):**
- Rotación de logs a fichero.
- Integración con servicios externos (Datadog, CloudWatch, etc.).
- Logging estructurado avanzado (JSON logs).
- Métricas y trazas (OpenTelemetry).

---

## 2. Alcance funcional
Este logger es **infraestructura técnica**, no funcionalidad de usuario.

Casos de uso:
- Registrar arranque del servidor y configuración aplicada.
- Registrar errores no controlados y excepciones.
- Registrar advertencias operativas.
- Registrar información de depuración en desarrollo.

---

## 3. Modelo de dominio

### 3.1 Tipos
**LogLevel**
- debug | info | warn | error

**LogContext**
- Diccionario clave-valor para metadatos contextuales.
- Ejemplo: `{ userId: '123', action: 'login' }`

### 3.2 Interfaz del puerto

```typescript
export type LogLevel = 'debug' | 'info' | 'warn' | 'error';
export type LogContext = Record<string, unknown>;

export interface Logger {
    debug(message: string, context?: LogContext): void;
    info(message: string, context?: LogContext): void;
    warn(message: string, context?: LogContext): void;
    error(message: string, context?: LogContext, error?: Error): void;
}
```

### 3.3 Invariantes
- El logger nunca lanza excepciones; absorbe errores internos.
- El logger nunca registra información sensible (passwords, tokens, hashes).
- Los métodos son síncronos para simplicidad de uso.

---

## 4. Adaptadores

### 4.1 ConsoleLogger (desarrollo)
- Escribe a `console.*` con formato legible.
- Incluye timestamp, nivel y contexto.
- Nivel mínimo configurable.

### 4.2 NoopLogger (tests)
- No produce salida.
- Permite tests silenciosos.

### 4.3 Evolución futura
- PinoLogger: logging estructurado JSON para producción.
- FileLogger: rotación a fichero.
- RemoteLogger: envío a servicios externos.

---

## 5. Integración

### 5.1 Composition Root
El logger se instancia en `composition/index.ts` según entorno:

| Entorno | Adaptador | Nivel mínimo |
|---------|-----------|--------------|
| development | ConsoleLogger | debug |
| production | ConsoleLogger | info |
| test | NoopLogger | - |

### 5.2 Inyección
El logger se inyecta como dependencia en:
- Casos de uso (opcional, para diagnóstico).
- Adaptadores de infraestructura.
- Servidor HTTP (arranque y errores).

---

## 6. Seguridad
- No registrar: passwords, tokens, hashes, datos personales sensibles.
- En caso de error, registrar solo el mensaje y stack, no payloads completos.

---

## 7. Decisiones de diseño (ADR-style)

### D1 — Logger como puerto
**Decisión:** definir Logger como interfaz en `application/ports`.
**Motivo:** permite inyección y testing, respeta Clean Architecture.
**Consecuencia:** cualquier capa puede usar el logger sin acoplarse a implementación.

### D2 — Métodos síncronos
**Decisión:** los métodos no devuelven Promise.
**Motivo:** simplicidad de uso; el logging no debe bloquear flujos.
**Consecuencia:** los adaptadores deben manejar I/O de forma no bloqueante si es necesario.

### D3 — Sin Result en firma
**Decisión:** los métodos devuelven void, no Result.
**Motivo:** el logging es fire-and-forget; los errores de logging no deben afectar el flujo.
**Consecuencia:** los adaptadores absorben errores internos.

### D4 — ConsoleLogger como adaptador inicial
**Decisión:** empezar con ConsoleLogger simple.
**Motivo:** mínima complejidad para v1; escalable a Pino/Winston después.
**Consecuencia:** se puede evolucionar sin cambiar el puerto ni los consumidores.

---

## 8. Estructura de archivos

```
src/
  application/
    ports/
      logger.ts
  infrastructure/
    adapters/
      logging/
        console-logger.ts
        noop-logger.ts
  composition/
    index.ts  (añadir wiring)
tests/
  infrastructure/
    adapters/
      logging/
        console-logger.test.ts
        noop-logger.test.ts
```

---

## 9. Riesgos y evolución futura
- Riesgo de logging excesivo en producción → definir política de niveles.
- Evolución: añadir PinoLogger para logs estructurados JSON.
- Evolución: integrar con Fastify logger existente como adaptador.
- Evolución: añadir correlationId por request para trazabilidad.
