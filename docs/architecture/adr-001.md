# ADR-0001: Arquitectura Base del Proyecto Gestiora — Monolito Modular con Clean Architecture (DDD)

## Estado

Aceptado

## Fecha

2026-01-26

## Contexto

Gestiora se encuentra en una **fase inicial (fase 1)** de desarrollo. El objetivo principal es construir una base sólida, mantenible y evolutiva sin introducir complejidad innecesaria desde el inicio.

El sistema debe permitir evolucionar hacia una arquitectura más modular en fases posteriores, sin requerir rediseños profundos.

Las necesidades actuales incluyen:

* Desarrollo rápido y controlado
* Claridad arquitectónica para el equipo
* Aislamiento del dominio frente a frameworks e infraestructura
* Preparación para escalabilidad futura

## Decisión

Se adopta una **arquitectura basada en Clean Architecture combinada con principios de Domain-Driven Design (DDD)**, implementada inicialmente como un **monolito no modularizado por bounded contexts**.

La modularización explícita por dominios se pospone a una **fase 2**, una vez que el dominio esté más estabilizado y validado.

## Arquitectura Elegida

### Tipo de sistema

* **Monolito** (despliegue único)
* **Estructura preparada para monolito modular**

### Estilo arquitectónico

* **Clean Architecture** como marco estructural
* **DDD** como guía de modelado del dominio

## Estructura de Carpetas Acordada

```
src/
  domain/
    entities/
    value-objects/
    events/
    errors/
  application/
    use-cases/
    ports/
    dto/
    errors/
  infrastructure/
    delivery/
      http/
    persistence/
    adapters/
  composition/
  shared/
  config/
test/
```

## Responsabilidades por Capa

### Domain

* Contiene el **modelo de negocio**
* Entidades, Value Objects, Domain Events y errores de dominio
* No depende de ninguna otra capa
* No contiene frameworks ni detalles técnicos

### Application

* Contiene los **casos de uso** del sistema
* Orquesta el flujo de negocio
* Define **puertos (interfaces outbound)** para dependencias externas
* Depende únicamente de `domain`
* Utiliza DTOs y errores propios de la capa

### Infrastructure

* Implementa los detalles técnicos
* Adaptadores de persistencia, HTTP, servicios externos, mensajería
* Puede depender de `application` y `domain`

### Composition

* Composition root para wiring de dependencias
* Configura adaptadores y casos de uso
* No contiene logica de dominio

### Shared

* Código transversal **no relacionado con el dominio de negocio**
* Utilidades técnicas, tipos base, helpers
* Uso restringido para evitar acoplamiento indebido

### Config

* Configuración del sistema
* Wiring / Dependency Injection
* Variables de entorno y bootstrap

## Reglas de Dependencia (Inviolables)

1. `domain` **no depende** de ninguna otra capa
2. `application` **puede depender** de `domain`, pero **nunca** de `infrastructure`
3. `infrastructure` **puede depender** de `application` y `domain`

Representación:

```
domain  <-  application  <-  infrastructure
```

## Decisiones Conscientes y Trade-offs

### Lo que se prioriza

* Simplicidad inicial
* Time-to-market
* Testabilidad
* Evolución controlada

### Lo que se pospone

* Microservicios
* Modularización estricta por bounded contexts
* Arquitecturas distribuidas complejas

Estas decisiones se consideran adecuadas para la fase actual del proyecto y **revisables** en el futuro.

## Evolución Esperada (Fase 2)

Cuando el dominio esté más maduro:

* Introducir `src/modules/<bounded-context>/...`
* Reubicar `domain/application/infrastructure` dentro de cada módulo
* Mantener las mismas reglas de dependencia

La estructura actual está diseñada para que esta transición sea mayormente organizativa.

## Consecuencias

### Positivas

* Arquitectura clara y comprensible
* Base sólida para crecimiento
* Menor riesgo técnico temprano

### Negativas

* Menor aislamiento entre dominios en fase 1
* Posible aumento temporal de dependencias internas si no se respeta disciplina

## Notas para Agentes de IA

Este ADR define:

* El **estilo arquitectónico oficial** del proyecto
* Las **reglas de dependencia**
* La **estructura base del código**

Cualquier generación de código, diseño o refactorización automática debe:

* Respetar estrictamente las reglas de Clean Architecture
* Mantener el dominio libre de dependencias técnicas
* Seguir la estructura aquí definida

## Revisión

Este ADR debe revisarse cuando:

* Se introduzca modularización por dominios
* Se evalúe adopción de microservicios
* Cambien de forma significativa los requisitos no funcionales
