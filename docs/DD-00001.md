# Design Doc — DD-00001 Autenticación, Autorización y Gestión de Sesiones (IAM Core)

## 1. Contexto y objetivo
Gestiora requiere un acceso **seguro por defecto** desde el día 1, con capacidad de:
- Autenticar usuarios (login/logout).
- Autorizar acciones por rol (RBAC básico).
- Gestionar sesiones revocables (refresh token persistido en servidor).
- Auditar acciones de seguridad.

**Objetivo principal:** habilitar un modelo de autenticación robusto para un backend API, compatible con evolución futura (más roles, permisos finos, MFA, SSO, etc.).

**Fuera de alcance (por ahora):**
- Recuperación de contraseña, cambio de contraseña por el usuario.
- MFA / 2FA.
- SSO (OIDC/SAML).
- Permisos por recurso (ABAC) o multi-tenant completo (si no está confirmado).

## 3. Modelo de dominio

### 3.1 Entidades principales
**Usuario**
- uuid
- mail (único)
- passwordHash
- estado: ACTIVO | INACTIVO | ELIMINADO
- lockedUntil (opcional)
- createdAt / updatedAt

**Rol**
- id
- nombre (Usuario | Administrador)

**Sesion**
- uuid
- userId
- refreshTokenHash
- status: ACTIVA | REVOCADA | EXPIRADA
- createdAt
- lastUsedAt
- expiresAt
- revokedAt (nullable)
- revokedBy (nullable: userId/adminId/system)
- ip (opcional)
- userAgent (opcional)

**IntentoLogin** (opcional como entidad, o como agregado de seguridad)
- userId/email
- ip
- timestamp
- resultado (OK/FAIL)
- reason (genérico)
- counterWindowKey (para rate limiting)

**EventoAuditoria**
- uuid
- actorUserId (nullable si no autenticado)
- action (LOGIN_SUCCESS, LOGIN_FAIL, LOGOUT, REFRESH, ACCESS_DENIED, USER_LOCKED, etc.)
- target (userId si aplica)
- metadata (ip, userAgent, requestId, etc.)
- createdAt

### 3.2 Invariantes y reglas
- Un `Usuario.mail` es único.
- Un `Usuario` INACTIVO o ELIMINADO **no puede** autenticarse.
- Un `RefreshToken` **nunca** se almacena en claro: solo `refreshTokenHash`.
- Una `Sesion` REVOCADA no puede refrescar.
- Un endpoint privado requiere access token válido.
- Un endpoint admin requiere rol Administrador.

## 4. Casos de uso (Application Layer)

> Nota: los nombres pueden mapearse a `UseCase` en Clean Architecture.

### UC-01 Login
**Input:** mail, password, (contexto: ip, userAgent)  
**Output:** accessToken, refreshToken, user profile básico (opcional), expiresIn  
**Reglas:**
- Validar credenciales sin filtrar información sensible.
- Verificar estado del usuario.
- Aplicar rate limit / anti fuerza bruta.
- Crear `Sesion` persistida y devolver tokens.
- Auditar: éxito / fallo.

**Errores:**
- `AUTH_INVALID_CREDENTIALS` (mensaje genérico)
- `AUTH_USER_DISABLED`
- `AUTH_USER_LOCKED`
- `RATE_LIMITED`

### UC-02 Logout
**Input:** refreshToken (o sessionId derivado), contexto  
**Output:** 204 / ok  
**Reglas:**
- Revocar sesión asociada al refresh.
- Inutilizar refresh token.
- Auditar.

**Errores:**
- si refresh inválido: responder 204 igualmente (idempotente) o 401 según criterio (recomendado: idempotente para logout).

### UC-03 Refresh Access Token
**Input:** refreshToken  
**Output:** nuevo accessToken (+ opcional: refreshToken nuevo si hay rotación)  
**Reglas:**
- Validar refresh (firma/estructura si aplica) + buscar hash en servidor.
- Comprobar estado sesión y expiración.
- Emitir nuevo access token.
- Auditar refresh.
- **Rotación opcional:** emitir refresh nuevo y revocar el anterior.

**Errores:**
- 401 si inválido / expirado / revocado.

### UC-04 Authorize Request (middleware / policy)
**Input:** accessToken, endpoint metadata  
**Output:** allow/deny + principal (userId, role)  
**Reglas:**
- Verificar firma y expiración del JWT.
- Enriquecer contexto con rol/claims.
- En endpoints admin: exigir rol Administrador.

**Errores:**
- 401 si no autenticado / token inválido
- 403 si autenticado sin permisos

### UC-05 Anti-Brute Force
**Input:** email+ip, timestamp  
**Output:** allow/deny  
**Reglas:**
- Limitar N intentos por ventana temporal.
- Opcional: bloqueo por usuario o por ip o combinado.
- Si bloquea usuario: set lockedUntil.

## 5. API / Interfaces

### 5.1 Endpoints propuestos
- `POST /auth/login`
  - body: `{ "email": "...", "password": "..." }`
  - response 200: `{ accessToken, refreshToken, expiresIn }`
- `POST /auth/logout`
  - body: `{ "refreshToken": "..." }` (o cookie)
  - response 204
- `POST /auth/refresh`
  - body: `{ "refreshToken": "..." }`
  - response 200: `{ accessToken, refreshToken?, expiresIn }`

### 5.2 Header / Tokens
- `Authorization: Bearer <accessToken>`
- Access token: JWT corto (10–15 min).
- Refresh token: largo (7–30 días), persistido como hash asociado a Sesion.

### 5.3 Dónde guardar el refresh token
Recomendación (backend-agnóstico):
- Si web: cookie httpOnly + secure + sameSite.
- Si CLI/otros: almacenamiento seguro local (keychain/secret manager), nunca en logs.

- `users.deletedAt` o `status=ELIMINADO`
- `users.status=INACTIVO`
- `sessions.status=REVOCADA|EXPIRADA`

## 7. Seguridad y permisos

### 7.1 Hash de contraseña
- Argon2id (preferido) o bcrypt (aceptable).
- Política mínima: longitud y complejidad configurable.
- Nunca almacenar passwords ni hashes en logs.

### 7.2 Refresh token (persistencia segura)
- Persistir hash: `hash(refreshToken)` (p.ej. SHA-256 + salt/pepper, o HMAC).
- Comparación en tiempo constante.
- Rotación opcional para mitigar robo.

### 7.3 Mitigación de fuerza bruta
- Rate limit por IP y por identidad (email normalizado).
- Bloqueo temporal tras N intentos fallidos.
- Respuestas genéricas siempre.

## 8. Observabilidad y auditoría

### 8.1 Eventos auditables mínimos
- LOGIN_SUCCESS / LOGIN_FAIL
- LOGOUT
- REFRESH_SUCCESS / REFRESH_FAIL
- ACCESS_DENIED (403)
- USER_LOCKED / USER_UNLOCKED
- SESSION_REVOKED (por admin o sistema)

### 8.2 Logging
- CorrelationId / RequestId por request.
- Nunca loggear: password, refresh token, access token completo.
- Loggear: userId (si existe), ip, userAgent (si aplica), resultado.

## 9. Decisiones de diseño (ADR-style)

### D1 — JWT corto + Refresh persistido y revocable
**Decisión:** Access token JWT corto, refresh token persistido como sesión revocable.  
**Motivo:** equilibrio entre rendimiento (JWT stateless) y seguridad (revocación real).  
**Consecuencia:** mantener tabla `sessions` y lógica de refresh/logout.

### D2 — RBAC básico con 2 roles
**Decisión:** roles Usuario/Administrador, enforcement en middleware/policies.  
**Consecuencia:** preparado para permisos más granulares sin romper API.

### D3 — Logout idempotente
**Decisión:** permitir logout aunque refresh no exista (o ya esté revocado).  
**Consecuencia:** UX estable, evita filtrar información sobre sesiones.

### D4 — Rotación de refresh (opcional)
**Decisión:** no obligatoria en v1, pero se deja diseñado.  
**Consecuencia:** menor complejidad inicial; se puede activar por configuración.

---

## 10. Riesgos y evolución futura

### Riesgos
- Implementación débil de rate limit → ataques de fuerza bruta.
- Gestión incorrecta de expiraciones y revocación → sesiones “fantasma”.
- Exposición accidental de tokens en logs.

### Evolución
- Permisos por recurso (documentos por empresa/usuario).
- MFA / 2FA.
- Recuperación de contraseña + políticas avanzadas.
- SSO (OIDC) para entornos corporativos.
- Multi-tenant formal (Empresa como boundary de acceso).
