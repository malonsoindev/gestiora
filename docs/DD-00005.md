# Design Doc — DD-00005 Extracción de Información mediante IA

## 1. Contexto y objetivo
Este módulo introduce la capacidad de **extraer automáticamente información estructurada** desde documentos (facturas) mediante IA.

Principios rectores:
- La IA actúa como **asistente**, nunca como fuente de verdad.
- El **documento original** sigue siendo la referencia principal (DD-00004).
- Los datos extraídos deben ser **revisables, corregibles y trazables**.
- La extracción se ejecuta de forma **asíncrona** (no bloqueante).

Objetivo:
- Reducir el trabajo manual de registro.
- Proveer datos estructurados para búsquedas y reporting.
- Preparar base para RAG (DD-00006).

**Fuera de alcance (v1):**
- Entrenamiento continuo / fine-tuning.
- OCR avanzado multi-idioma (salvo que se defina).
- Aprendizaje por correcciones del usuario.
- Automatizar acciones contables (solo extracción).

## 2. Alcance funcional
Cubre **Épica E — Extracción IA**:

- US-E01 Extracción automática al subir documento
- US-E02 Datos mínimos extraídos

- US-E04 Revisión y validación

- US-E07 Trazabilidad/explicabilidad

### Roles implicados
- Usuario (revisa/valida/corrige)
- Sistema (ejecuta extracción)

## 3. Modelo de dominio

### 3.1 Entidades / Value Objects

**ExtractionJob**
- uuid
- documentId
- status: PENDIENTE | EN_PROCESO | COMPLETADO | FALLIDO
- trigger: AUTO_ON_UPLOAD | MANUAL_REPROCESS
- attempts
- createdAt
- startedAt
- finishedAt
- errorCode (nullable)
- errorDetail (nullable, limitado)

**ExtractedField** (Value Object o entidad)
- fieldName (e.g. invoiceNumber, invoiceDate, total, vat, supplierName, lineItems)
- value (tipado según campo)
- source: AI
- confidence (0..1 o LOW/MED/HIGH)
- status: PROPUESTO | CONFIRMADO | RECHAZADO
- evidenceRef (nullable)  -> referencia a evidencias (texto/posiciones)

**ExtractionResult**
- documentId
- extractedFields[] (colección de ExtractedField)
- version (incremental por reprocesado)
- createdAt

**Proveedor (borrador)** (ver DD-00003)
- status: BORRADOR
- datos parciales detectados
- flags: PENDING_REVIEW

**Evidence**
- uuid
- documentId
- extractionResultVersion
- type: TEXT_SNIPPET | PAGE_REGION | TOKEN_SPAN
- payload: texto (limitado) o coordenadas (si aplica)
- pageNumber (nullable)
- createdAt

> Nota: La evidencia puede ser un módulo mínimo en v1: guardando solo fragmentos de texto relevantes.
> Si más adelante se usa OCR + bounding boxes, se amplía.

---

### 3.2 Agregados
- **Documento** sigue siendo el Aggregate Root de la vida del documento.
- La extracción se modela como agregado independiente:
  - `ExtractionJob` + `ExtractionResult` (por documento y versión).
- **Proveedor** sigue siendo agregado propio, pero puede crearse un **BORRADOR** desde extracción.

---

### 3.3 Invariantes de dominio
- La extracción no debe bloquear la disponibilidad del documento.
- Un `ExtractionJob` COMPLETADO no se modifica; un reprocesado crea nueva versión.
- Los campos confirmados por el usuario deben preservarse:
  - un reprocesado no puede sobrescribir datos CONFIRMADOS sin intervención explícita.
- Si el proveedor detectado no existe:
  - se crea proveedor BORRADOR y se asocia al documento.
- El sistema debe exponer qué campos son AI vs manual y su estado.

## 4. Casos de uso (Application Layer)

### UC-01 Disparar extracción al subir documento
**Trigger:** evento `DOCUMENT_UPLOADED`  
**Input:** documentId  
**Output:** jobId  

**Reglas:**
- Crear `ExtractionJob` status PENDIENTE.
- Encolar job en worker/cola.
- Documento accesible inmediatamente.
- Auditar evento de extracción iniciada.

---

### UC-02 Procesar extracción (worker)
**Input:** jobId  
**Output:** `ExtractionResult` version N, evidencias, actualización de status  

**Pasos:**
1. Marcar job EN_PROCESO.
2. Recuperar PDF / texto.
3. Ejecutar pipeline de extracción:
   - (a) OCR si procede
   - (b) parsing layout / texto
   - (c) modelo IA para extracción estructurada
4. Generar `ExtractedField[]` con confidence.
5. Persistir `ExtractionResult` (versión N).
6. Generar `Evidence[]` asociadas.
7. Resolver proveedor:
   - buscar match por CIF o razón social normalizada
   - si no existe → crear proveedor BORRADOR
8. Marcar job COMPLETADO.

**Errores:**
- Si falla → job FALLIDO con errorCode, attempts++ y política de reintentos.

---

### UC-03 Consultar resultados de extracción
**Input:** documentId  
**Output:** extracción actual + estados por campo  

**Reglas:**
- Mostrar campos extraídos y su status.
- Indicar claramente qué campos son AI vs manual.
- Incluir evidencias si existen.

---

### UC-04 Validar/corregir campos extraídos
**Input:** documentId, cambios {fieldName -> value + status}  
**Output:** ok  

**Reglas:**
- Campo confirmado pasa a CONFIRMADO.
- Campo corregido pasa a CONFIRMADO con valor usuario.
- Campo rechazado pasa a RECHAZADO.
- No sobrescribir manual sin confirmación explícita.
- Auditar acción.

---

### UC-05 Reprocesar extracción
**Input:** documentId  
**Output:** nuevo jobId  

**Reglas:**
- Crear nuevo `ExtractionJob` trigger MANUAL_REPROCESS.
- Nueva versión en `ExtractionResult`.
- Respetar campos confirmados (no overwrite).
- Auditar reprocesado.

---

### UC-06 Gestionar estados y notificación (mínimo)
**Input:** documentId / jobId  
**Output:** status actualizado  

**Reglas:**
- Estados: PENDIENTE / EN_PROCESO / COMPLETADO / FALLIDO.
- El frontend/cliente puede consultar por polling.
- (Opcional) emitir evento cuando finaliza.

---

## 5. Arquitectura y flujo asíncrono

### 5.1 Componentes
- API Backend (Gestiora)
- Cola de trabajos (RabbitMQ/SQS/Redis Streams/etc.)
- Worker de extracción
- Storage de archivos (S3/MinIO/etc.)
- Persistencia (DB)

### 5.2 Flujo
1. Usuario sube PDF → se crea Documento (DD-00004).
2. Se emite evento `DOCUMENT_UPLOADED`.
3. Se crea `ExtractionJob` y se encola.
4. Worker procesa y persiste `ExtractionResult`.
5. Documento queda con estado de extracción actualizado.
6. Usuario revisa/corrige → se consolidan campos confirmados.

---

## 6. API / Interfaces

### Endpoints propuestos
- `GET    /documents/{id}/extraction`  
  Devuelve estado del job actual + extracción versión activa.
- `POST   /documents/{id}/extraction/reprocess`  
  Dispara reprocesado.
- `PUT    /documents/{id}/extraction/fields`  
  Permite confirmar/corregir/rechazar campos.
- `GET    /documents/{id}/extraction/evidence` (opcional v1)  
  Devuelve evidencias por campo.

### Contrato de datos (orientativo)
- Campo:
  - `name`
  - `value`
  - `confidence`
  - `status`
  - `source=AI|MANUAL`
  - `evidenceIds[]`

## 7. Persistencia

### 7.1 Tablas / colecciones
- `extraction_jobs`
- `extraction_results` (por documentId + version)
- `extracted_fields` (normalizada o JSON según DB)
- `evidences` (opcional en v1, recomendable)
- `providers` (incluye BORRADOR)
- `documents` (referencia a extraction status/version)

### 7.2 Estrategia de versionado
- `extraction_results.version` incremental por documento.
- La versión activa puede ser la última COMPLETADA.
- Mantener historial para trazabilidad.

---

## 8. Seguridad, privacidad y cumplimiento

- Acceso a extracción y evidencias solo para usuarios con acceso al documento.
- Minimizar datos sensibles en:
  - logs
  - errorDetail
  - evidencias (fragmentos limitados, nunca el PDF completo)
- Sanitizar outputs del modelo:
  - validar tipos y rangos (fechas, importes)
- No exponer prompts internos ni tokens.

---

## 9. Observabilidad y auditoría

### Eventos mínimos
- EXTRACTION_JOB_CREATED
- EXTRACTION_JOB_STARTED
- EXTRACTION_JOB_COMPLETED
- EXTRACTION_JOB_FAILED
- EXTRACTION_FIELDS_REVIEWED (confirmados/corregidos)
- EXTRACTION_REPROCESS_REQUESTED
- DRAFT_PROVIDER_CREATED (si aplica)

### Métricas recomendadas
- tiempo medio de extracción
- ratio de fallos
- número de reprocesados
- porcentaje de campos confirmados sin cambios (calidad IA)

## 10. Decisiones de diseño (ADR-style)

### D1 — Extracción asíncrona no bloqueante
**Decisión:** siempre asíncrona y el documento es usable desde el inicio.  
**Consecuencia:** UX estable y sistema escalable.

### D2 — Datos extraídos con estados por campo
**Decisión:** cada campo tiene status PROPUESTO/CONFIRMADO/RECHAZADO.  
**Consecuencia:** evita tratar IA como verdad; mejora trazabilidad.

### D3 — Reprocesado versionado
**Decisión:** reprocesado genera nueva versión, no muta la anterior.  
**Consecuencia:** auditoría completa y comparabilidad.

### D4 — Borrador de proveedor si no existe
**Decisión:** crear proveedor BORRADOR con revisión pendiente.  
**Consecuencia:** evita bloquear la asociación y facilita flujo de negocio.

---

## 11. Riesgos y evolución futura

### Riesgos
- Costes y latencia de extracción.
- Alucinación de la IA si no se valida/sanitiza.
- Evidencias insuficientes para justificar campos.
- Duplicidad de proveedores por matching débil.

### Evolución
- OCR con bounding boxes para evidencias visuales.
- Matching probabilístico de proveedores con score + UI de resolución.
- Aprendizaje por correcciones (human-in-the-loop).
- Extracción de líneas (movimientos) completa.
- Motor de reglas para validación fiscal (IVA, totales, etc.).
