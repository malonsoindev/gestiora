# Design Doc — DD-00007 Creacion Manual de Facturas sin PDF y Vinculacion Posterior

## 1. Contexto y objetivo
Gestiora necesita soportar dos vias de generacion de facturas:

- **Via automatica (PDF + IA):** un endpoint recibe solo el PDF, un agente de IA (Genkit)
  extrae la informacion y crea la factura vinculada al archivo.
- **Via manual (usuario):** el usuario crea la factura con cabecera y movimientos
  en una sola llamada, con PDF opcional (puede adjuntarse despues).

El objetivo de este documento es definir el flujo manual sin PDF y la vinculacion
posterior del archivo, manteniendo coherencia con DD-00004 (facturas) y DD-00005 (extraccion).

---

## 2. Alcance funcional
Extiende la Epica D con historias adicionales:

- **US-D09** Crear documento manual (cabecera + movimientos) sin PDF.
- **US-D10** Adjuntar PDF a documento manual existente.
- **US-D11** Actualizar cabecera y movimientos de factura manual.

La via automatica (PDF + IA) se mantiene en DD-00005.

---

## 3. Modelo de dominio

### 3.1 Entidades y value objects

**Factura** (Aggregate Root)
- uuid
- providerId
- fileRefId (opcional)
- status: DRAFT | ACTIVO | ELIMINADO
- numeroFactura
- fechaOperacion
- fechaVencimiento
- baseImponible
- iva
- total
- createdAt
- updatedAt
- deletedAt (nullable)

**Movimiento**
- uuid
- facturaId
- concepto
- cantidad
- precio
- baseImponible
- iva
- total

**FileRef**
- storageKey
- filename
- mimeType
- sizeBytes
- checksum

---

### 3.2 Invariantes
- Una Factura **DRAFT** puede existir sin FileRef.
- Una Factura **ACTIVO** debe tener FileRef valido.
- Una Factura **ELIMINADO** no aparece en listados operativos.
- La factura manual (cabecera + movimientos) se crea en una sola llamada.
- Las facturas manuales forman parte de la fuente de conocimiento, incluso sin PDF
  (al menos por sus metadatos y movimientos).

---

## 4. Casos de uso (Application Layer)

### UC-01 Crear factura manual
**Input:** providerId, cabecera de factura, movimientos[]
**Output:** invoiceId

**Reglas:**
- Usuario autenticado.
- Proveedor debe estar ACTIVO.
- Crear Factura en estado DRAFT (sin FileRef).
- Persistir Factura y Movimientos en una sola transaccion.
- Auditar creacion.

**Errores:**
- `PROVIDER_INACTIVE`
- `VALIDATION_ERROR`

---

### UC-02 Adjuntar PDF a factura manual
**Input:** invoiceId, archivo PDF
**Output:** ok

**Reglas:**
- Factura debe existir y no estar ELIMINADO.
- Validar formato/tamano del archivo.
- Crear FileRef y asociarlo a la Factura.
- Transicionar status a ACTIVO.
- Auditar vinculacion.

**Errores:**
- `INVOICE_NOT_FOUND`
- `INVALID_FILE_FORMAT`
- `FILE_TOO_LARGE`

---

### UC-03 Actualizar factura manual
**Input:** invoiceId, cabecera de factura, movimientos[]
**Output:** ok

**Reglas:**
- Factura debe existir y no estar ELIMINADO.
- Actualizar cabecera y movimientos como conjunto.
- Mantener consistencia entre totales y lineas.
- Auditar actualizacion.

---

## 5. API / Interfaces

### 5.1 Endpoints propuestos (manual)
- `POST /documents/manual`
  - body: JSON con cabecera + movimientos
  - response 201: `{ invoiceId }`

- `PUT /documents/{invoiceId}/file`
  - body: multipart/form-data con PDF
  - response 200 o 204

El endpoint existente `POST /documents` queda reservado para la via automatica
(PDF + IA) y recibe **solo** el archivo.

---

## 6. Persistencia
- `invoices` incluye `status=DRAFT` y `file_ref_id` opcional.
- `movements` se persisten en la creacion manual.
- `file_refs` se crea al adjuntar PDF.

---

## 7. Seguridad y permisos
- Todos los endpoints requieren autenticacion.
- Validar que el usuario puede operar sobre el proveedor/factura.

---

## 8. Observabilidad y auditoria
Eventos minimos:
- INVOICE_MANUAL_CREATED
- INVOICE_FILE_ATTACHED
- INVOICE_MANUAL_UPDATED
- MOVEMENTS_UPDATED

---

## 9. Decisiones de diseno (ADR-style)

### D1 — Doble via de creacion
**Decision:** mantener vias separadas: automatica (PDF + IA) y manual (cabecera + movimientos).
**Consecuencia:** endpoints y flujos distintos, pero ambos alimentan la misma base de conocimiento.

### D2 — Estado DRAFT para facturas sin PDF
**Decision:** permitir Factura DRAFT sin FileRef y exigir FileRef solo en ACTIVO.
**Consecuencia:** coherencia con el principio de documento fuente de verdad y permite
creacion manual sin archivo.

### D3 — Movimientos como parte del input manual
**Decision:** cabecera y movimientos se crean en una sola llamada.
**Consecuencia:** consistencia transaccional y menos pasos de usuario.

---

## 10. Riesgos y evolucion futura
- Riesgo de facturas sin PDF durante mucho tiempo (definir politica de adjunto).
- Necesidad de reglas de validacion fiscal mas estrictas.
- Posible extension a versionado de documentos y multiples archivos.
