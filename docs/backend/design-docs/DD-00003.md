# Design Doc — DD-00003 Gestión de Proveedores

## 1. Contexto y objetivo
El proveedor es una **entidad central del dominio Gestiora**.  
Toda factura o documento debe estar asociado a un proveedor, ya sea creado manualmente o detectado posteriormente por procesos de IA.

El objetivo de este módulo es:
- Permitir la **gestión manual completa de proveedores**.
- Garantizar la **no duplicidad** y la coherencia de datos.
- Mantener trazabilidad histórica aunque el proveedor deje de estar activo.
- Servir como base para:
  - gestión documental,
  - extracción automática por IA,
  - búsqueda avanzada (RAG),
  - integraciones futuras (ERP, contabilidad, etc.).

**Fuera de alcance (v1):**
- Fusión de proveedores.
- Enriquecimiento automático externo.
- Creación automática por IA (solo borradores en DD-00005).
- Sincronización con sistemas externos.

## 3. Modelo de dominio

### 3.1 Entidad principal

**Proveedor**
- uuid
- razonSocial
- cif (opcional pero recomendado)
- direccion
- poblacion
- provincia
- pais
- status: ACTIVO | INACTIVO | ELIMINADO | BORRADOR
- createdAt
- updatedAt
- deletedAt (nullable)

> Nota: el estado **BORRADOR** se introduce para compatibilidad futura con IA (DD-00005).

---

### 3.2 Agregado
- **Proveedor** es un **Aggregate Root**.
- Todas las operaciones sobre proveedores pasan por él.
- Documentos y facturas **referencian** al proveedor, pero no lo modifican.

---

### 3.3 Invariantes de dominio
- Un proveedor ACTIVO puede asociarse a nuevos documentos.
- Un proveedor INACTIVO:
  - no puede asociarse a nuevos documentos,
  - sigue siendo visible en histórico.
- Un proveedor ELIMINADO:
  - no aparece en listados operativos,
  - mantiene referencias históricas.
- No se permite borrado físico.
- La duplicidad debe prevenirse según reglas configurables.

---

## 4. Casos de uso (Application Layer)

### UC-01 Crear proveedor
**Input:** datos básicos del proveedor  
**Output:** proveedorId  

**Reglas:**
- Usuario autenticado.
- Validar reglas de duplicidad antes de persistir.
- Crear proveedor como ACTIVO por defecto.
- Auditar creación.

**Errores:**
- `PROVIDER_ALREADY_EXISTS`
- `VALIDATION_ERROR`

---

### UC-02 Listar proveedores
**Input:** filtros (estado, texto), paginación  
**Output:** lista de proveedores  

**Reglas:**
- Excluir ELIMINADOS por defecto.
- Permitir filtros por estado.
- No exponer datos sensibles innecesarios.

---

### UC-03 Consultar proveedor
**Input:** proveedorId  
**Output:** detalle completo  

**Reglas:**
- 404 si no existe o está eliminado.
- Accesible aunque esté INACTIVO.

---

### UC-04 Modificar proveedor
**Input:** proveedorId, campos modificables  
**Campos permitidos:**
- datos de identificación
- dirección
- país / provincia

**Reglas:**
- No modificar uuid.
- Revalidar duplicidad si se modifican campos clave.
- Auditar cambios.

---

### UC-05 Activar / desactivar proveedor
**Input:** proveedorId, nuevo estado  
**Output:** ok  

**Reglas:**
- Desactivar:
  - impide nuevas asociaciones.
- Activar:
  - vuelve a permitir asociaciones.
- No aplicable a proveedores ELIMINADOS.

---

### UC-06 Eliminación lógica
**Input:** proveedorId  
**Output:** ok  

**Reglas:**
- Marcar como ELIMINADO.
- Excluir de listados operativos.
- Mantener referencias históricas.
- No reversible.

---

### UC-07 Validación de duplicados
**Input:** datos normalizados del proveedor  
**Output:** ok / conflicto  

**Reglas posibles (configurables):**
- CIF exacto.
- Razón social normalizada.
- Combinación CIF + país.

## 7. Seguridad y control de acceso
- Acceso solo para usuarios autenticados.
- No se permite modificación indirecta desde otros dominios.
- Integridad referencial con documentos/facturas.

---

## 8. Observabilidad y auditoría

### Eventos mínimos
- PROVIDER_CREATED
- PROVIDER_UPDATED
- PROVIDER_ACTIVATED
- PROVIDER_DEACTIVATED
- PROVIDER_DELETED
- PROVIDER_DUPLICATE_DETECTED

Cada evento incluye:
- actorUserId
- providerId
- timestamp
- resultado

---

## 9. Decisiones de diseño (ADR-style)

### D1 — Proveedor como Aggregate Root
**Decisión:** el proveedor controla su consistencia.  
**Consecuencia:** otros dominios solo lo referencian.

### D2 — Soft delete obligatorio
**Decisión:** nunca borrar proveedores físicamente.  
**Consecuencia:** histórico íntegro y trazabilidad.

### D3 — Estado BORRADOR previsto
**Decisión:** incluir estado aunque no se use aún.  
**Consecuencia:** transición limpia hacia IA sin refactor.

---

## 10. Riesgos y evolución futura

### Riesgos
- Reglas de duplicidad demasiado laxas o estrictas.
- Proveedores mal normalizados.
- Falta de tooling para limpieza de datos.

### Evolión futura
- Fusión de proveedores.
- Enriquecimiento automático.
- Proveedores multi-empresa.
- Sincronización con ERP.
