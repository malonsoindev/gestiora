# Design Doc — DD-00006 Búsqueda y Consultas en Lenguaje Natural (RAG)

## 1. Contexto y objetivo
Este módulo introduce capacidades de **búsqueda avanzada y consultas en lenguaje natural** sobre los documentos y datos de Gestiora mediante un enfoque de **Retrieval-Augmented Generation (RAG)**.

El sistema debe permitir que un usuario:
- Formule preguntas en lenguaje natural.
- Obtenga respuestas basadas **exclusivamente** en documentos accesibles.
- Reciba **referencias claras** a los documentos fuente.
- Acceda directamente al documento original.

Principios clave:
- **Precisión > creatividad**
- **Trazabilidad obligatoria**
- **Cero alucinaciones**
- **Respeto estricto de permisos**

**Fuera de alcance (v1):**
- Chat conversacional persistente.
- Memoria de contexto entre consultas.
- Escritura o modificación de datos vía lenguaje natural.
- Acceso directo del modelo a la base de datos.

## 3. Arquitectura RAG (alto nivel)

### 3.1 Componentes
- API Backend Gestiora
- Motor de recuperación (retriever)
- Índice vectorial / híbrido
- Motor de generación (LLM)
- Sistema de permisos y filtros
- Sistema de evidencias

### 3.2 Principio fundamental
> El **modelo generativo nunca consulta directamente la base de datos**.  
> Solo recibe contexto previamente recuperado y filtrado.

---

## 4. Modelo conceptual

### 4.1 Entidades / Value Objects

**SearchQuery**
- uuid
- userId
- originalQuery
- normalizedQuery
- filtersDetected (proveedor, fechas, importes, etc.)
- createdAt

**RetrievedDocument**
- documentId
- score
- snippets[] (fragmentos relevantes)
- metadata (proveedor, fechas, importes)

**SearchResponse**
- queryId
- answer (texto generado)
- references[] (documentId + snippetIds)
- confidence (opcional, heurístico)
- createdAt

**Snippet / Evidence**
- uuid
- documentId
- text (limitado)
- pageNumber (nullable)
- source: OCR | TEXT | EXTRACTION
- createdAt

## 5. Flujo RAG

### 5.1 Flujo principal
1. Usuario envía consulta en lenguaje natural.
2. Normalización y parsing de la consulta.
3. Detección de filtros implícitos:
   - proveedor
   - fechas
   - importes
4. Recuperación de documentos candidatos:
   - solo documentos accesibles por el usuario
5. Selección de fragmentos relevantes (snippets).
6. Construcción de contexto controlado.
7. Generación de respuesta por el modelo.
8. Validación post-generación.
9. Respuesta con referencias.

---

## 6. Casos de uso (Application Layer)

### UC-01 Procesar consulta
**Input:** texto libre  
**Output:** respuesta + referencias  

**Reglas:**
- Usuario autenticado.
- Normalizar consulta (lowercase, idioma, fechas).
- Detectar ambigüedad.
- Auditar consulta.

**Errores:**
- `UNAUTHORIZED`
- `QUERY_TOO_AMBIGUOUS`

---

### UC-02 Recuperar documentos relevantes
**Input:** consulta normalizada + filtros  
**Output:** lista de `RetrievedDocument`  

**Reglas:**
- Aplicar filtros de permisos **antes** del retrieval.
- Limitar número de documentos (top-k).
- Priorizar documentos con metadatos confirmados.

---

### UC-03 Generar respuesta
**Input:** contexto limitado  
**Output:** texto generado + referencias  

**Reglas:**
- El modelo solo puede usar el contexto dado.
- Forzar formato de respuesta con citas.
- Si no hay contexto suficiente → respuesta negativa explícita.

---

### UC-04 Gestionar ambigüedad
**Input:** consulta  
**Output:** solicitud de aclaración o respuesta parcial  

**Reglas:**
- Detectar múltiples interpretaciones.
- Priorizar pedir aclaración frente a inventar.

---

### UC-05 Acceso a documentos desde resultados
**Input:** documentId  
**Output:** acceso al documento  

**Reglas:**
- Revalidar permisos.
- Reutilizar lógica de DD-00004.

---

## 7. Indexación y recuperación

### 7.1 Qué se indexa
- Texto del documento (OCR / texto plano).
- Campos estructurados confirmados:
  - proveedor
  - fechas
  - importes
- Fragmentos relevantes (snippets).

### 7.2 Estrategia de indexación
- Índice híbrido:
  - vectorial (semántico)
  - filtros estructurados (metadata)
- Reindexación cuando:
  - se confirman datos IA
  - se corrigen metadatos
  - se elimina un documento

---

## 8. Control de calidad y anti-alucinación

### 8.1 Medidas preventivas
- Contexto limitado y explícito.
- Prompt restrictivo:
  - “Responde solo con la información proporcionada.”
- No permitir inferencias externas.

### 8.2 Validación post-generación
- Verificar que todas las afirmaciones tienen referencia.
- Si no:
  - truncar respuesta
  - devolver mensaje de insuficiencia de datos.

## 10. Seguridad y permisos

- Todas las consultas requieren autenticación.
- El retrieval filtra por:
  - usuario
  - empresa (si aplica)
- El modelo nunca ve documentos no autorizados.
- No se almacena contenido completo sensible en logs.

## 11. Observabilidad y auditoría

### Eventos mínimos
- SEARCH_QUERY_RECEIVED
- SEARCH_DOCUMENTS_RETRIEVED
- SEARCH_RESPONSE_GENERATED
- SEARCH_RESPONSE_PARTIAL
- SEARCH_NO_ANSWER

### Auditoría (admin)
- userId
- timestamp
- tipo de consulta
- número de documentos usados
- no almacenar texto completo si contiene datos sensibles (configurable)

---

## 12. Decisiones de diseño (ADR-style)

### D1 — RAG estricto sin acceso a BD
**Decisión:** el modelo solo recibe contexto recuperado.  
**Consecuencia:** menor riesgo de alucinaciones.

### D2 — Respuestas siempre con referencias
**Decisión:** sin referencias → sin respuesta.  
**Consecuencia:** mayor confianza y verificabilidad.

### D3 — Gestión explícita de ambigüedad
**Decisión:** pedir aclaración antes que inferir.  
**Consecuencia:** UX más segura, menos “magia”.

---

## 13. Riesgos y evolución futura

### Riesgos
- Respuestas demasiado conservadoras.
- Índices mal sincronizados.
- Costes de embeddings y generación.
- Consultas ambiguas frecuentes.

### Evolución
- Chat conversacional con contexto acotado.
- Explicaciones visuales (highlight en PDF).
- Feedback del usuario sobre utilidad.
- Ranking adaptativo de documentos.
- Multilingüe.
